# Lab APS Terminology

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Draft

---

# 1. Purpose

This document defines the common terminology used throughout the Lab APS project.

All project documentation, database design, API specifications and source code shall follow these definitions.

When a term is defined in this document, all subsequent documents shall use the same definition.

---

# 2. Core Concepts

## 2.1 Plan

A **Plan** is the highest-level planning object.

A Plan represents a complete scheduling result for a specific planning period.

A Plan contains:

* Planning Horizon
* Production Demand
* Workflow Instances
* Schedule
* Material Forecast
* KPIs

Example

```text
Week 32 Production Plan
```

A Plan may have multiple versions.

---

## 2.2 Planning Horizon

The time range covered by a Plan.

Examples:

* One Week
* Two Weeks
* One Month

Version 1.0 supports one-week planning.

---

## 2.3 Demand

Demand represents the production requirement provided by the Principal Investigator (PI).

Demand answers only one question:

> What needs to be completed?

Example

| Project | Required Runs |
| ------- | ------------: |
| FV      |             6 |
| 96 OPA  |            12 |
| 384 PNG |            20 |

Demand does **not** specify:

* Equipment
* Operator
* Schedule

---

## 2.4 Project

A Project represents a laboratory testing service.

Examples

* FV
* 96 OPA
* 384 PNG
* AZ RSV
* DiLA

A Project is **not** directly scheduled.

Projects are converted into Workflow Instances.

---

## 2.5 Workflow Template

A Workflow Template defines the standard execution process of a Project.

Example

384 PNG

```text
SMDP
   │
   ▼
Output Plate
   │
   ▼
SAP
```

Workflow Templates are reusable and configurable.

---

## 2.6 Workflow Instance

A Workflow Instance is created from a Workflow Template for a specific production demand.

Example

```text
Workflow Template

384 PNG

↓

Workflow Instance

PNG-2026-001
```

Every Workflow Instance has its own execution status.

---

## 2.7 Operation

An Operation is the smallest executable scheduling unit.

Examples

* FV
* SMDP
* SAP
* CP
* SP

The Scheduler allocates Operations.

Operations define:

* Duration
* Required Capability
* Required Skill
* Required Resources
* Dependencies

---

## 2.8 Dependency

A Dependency defines execution order.

Example

```text
SMDP

↓

SAP
```

SAP cannot start until SMDP finishes.

Dependencies are generated by the Workflow Engine.

---

# 3. Resource Concepts

Only resources that affect scheduling are considered Scheduling Resources.

---

## 3.1 Staff

A laboratory operator capable of executing Operations.

Staff properties include:

* Skills
* Working Calendar
* Leave
* Status

---

## 3.2 Equipment

A laboratory instrument capable of executing Operations.

Examples

* SU-HM-01
* SU-HM-02
* SU-HM-09
* SU-HM-10

Equipment properties include:

* Capability
* Status
* Maintenance
* FV Qualification

---

## 3.3 Capability

Capability describes what an Equipment can perform.

Examples

* 96 Head
* 384 Head
* 16 Channel
* 14 Channel
* iSWAP

Operations require Capabilities.

Scheduling matches:

Required Capability

↓

Equipment Capability

---

## 3.4 Skill

Skill describes what a Staff member is qualified to perform.

Examples

* FV
* OPA
* PNG
* AZ RSV
* DiLA

Scheduling matches:

Required Skill

↓

Staff Skill

---

## 3.5 Intermediate Resource

An Intermediate Resource is produced by one Operation and consumed by another.

Example

```text
SMDP

↓

Output Plate

↓

SAP
```

Intermediate Resources participate in scheduling.

Properties include:

* Creation Time
* Shelf Life
* Stamp Out Count

---

# 4. Material Concepts

Materials do **not** participate in scheduling optimization.

Materials are calculated after scheduling.

---

## 4.1 Consumable

A disposable laboratory material.

Examples

* Tips
* Deepwell Block
* Plate
* Seal

Consumables are forecast after scheduling.

---

## 4.2 Reagent

A laboratory chemical or kit required by an Operation.

Current Version

Only FV requires reagent management.

Reagents are treated as forecast materials.

---

## 4.3 Material BOM

Material Bill of Materials defines expected material consumption for each Operation.

Example

384 PNG SMDP

* 384 Tips
* Deepwell Block

Material BOM is maintained in Workflow Templates.

---

# 5. Scheduling Concepts

## 5.1 Assignment

Assignment represents the scheduling result.

Assignment binds:

* Operation
* Staff
* Equipment
* Shift

Example

```text
PNG001-SMDP

↓

Tom

↓

HM09

↓

Monday Shift 1
```

---

## 5.2 Shift

A Shift is the smallest scheduling time unit.

Current supported shifts include:

Normal Shift

09:00 – 17:00

or

Shift 1

06:00 – 14:00

Shift 2

14:00 – 22:00

---

## 5.3 Schedule

A Schedule is the complete collection of Assignments generated by the Solver.

A Schedule belongs to exactly one Plan.

---

## 5.4 Publish

Publishing makes a Plan available for laboratory execution.

Only published Plans may be executed.

---

# 6. Forecast Concepts

## 6.1 Material Forecast

Material Forecast predicts laboratory material consumption after scheduling.

Outputs include:

* Daily Consumption
* Weekly Consumption
* Material Summary

---

## 6.2 Inventory Check

Inventory Check compares forecast consumption against available inventory.

Inventory management is outside the scope of Lab APS.

Lab APS only reports shortages.

---

# 7. System Concepts

## 7.1 Workflow Engine

Responsible for converting Workflow Templates into executable Workflow Instances and Operations.

---

## 7.2 Planning Engine

Coordinates planning activities.

Responsibilities include:

* Demand Processing
* Workflow Generation
* Schedule Generation
* Forecast Generation

---

## 7.3 Scheduling Engine

Responsible for optimization.

Functions include:

* Variable Generation
* Constraint Construction
* Objective Definition
* Solver Execution

---

## 7.4 Material Calculator

Calculates expected material consumption based on the generated Schedule.

---

## 7.5 Solver Adapter

Encapsulates OR-Tools implementation.

Business modules must communicate only with the Solver Adapter.

The Solver implementation may be replaced without changing business modules.

---

# 8. Naming Rules

The following terminology shall be used consistently throughout the project.

| Preferred Term        | Do Not Use          |
| --------------------- | ------------------- |
| Plan                  | Schedule Plan       |
| Workflow Template     | Process             |
| Workflow Instance     | Run                 |
| Operation             | Step / Task (混用)    |
| Assignment            | Allocation          |
| Capability            | Equipment Skill     |
| Skill                 | Operator Capability |
| Material Forecast     | Inventory Planning  |
| Intermediate Resource | Temporary Plate     |
| Publish               | Confirm Schedule    |
