# Development Log

## Milestone M10.1 — Token Authentication & Role Authorization

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Complete

**Owner:** Development Team

**Completed:** 2026-07-08

---

# 1. Objective

Introduce authentication and role-based authorization across the REST API, wired
through the Composition Root. Decision recorded in
[ADR-013](../04_ADR/ADR-013-Token-Authentication.md).

---

# 2. Scope

- `User` + `ApiToken` (ORM; only token hashes stored) and a `Role` enum
  (administrator, production_lm).
- `AuthService`: create a user with a token; resolve a bearer token to the
  current user.
- A `before_request` guard resolves `Authorization: Bearer <token>` to the
  current user (health is public); a `require_role` decorator gates actions.
- Auth errors (401 UNAUTHORIZED, 403 FORBIDDEN) use the standard envelope.
- API: `GET /api/v1/auth/whoami`; `POST /api/v1/users` (administrator only).
- Config flag `auth_enabled` (env `AUTH_ENABLED`); enabled by default. Business
  tests run with it disabled; identity tests enable it.
- Alembic migration for app_user and api_token.

# 4. Out of Scope

- Username/password login and sessions (deferred; see ADR-013 alternatives).
- Token rotation/revocation UI; per-endpoint fine-grained permissions beyond the
  two roles.
- Applying `require_role` to every existing business endpoint (authentication is
  enforced globally; role gating currently applies to user management — role
  gating of planning/execution actions is a follow-up).

---

# 5. Acceptance Criteria

- Unauthenticated requests to business endpoints return 401; health is public.
- A valid token authenticates; an invalid token is rejected.
- Administrators can create users; non-administrators get 403.
- Newly created users can authenticate with their token.

# 8. Completion Record

**Date:** 2026-07-08

Authentication is centralized in one guard; the domain, engines and solver never
see tokens or Flask state. The verification seam (resolve token -> user) is
isolated in `AuthService`, so a later switch to JWT does not touch endpoints. The
`auth_enabled` flag keeps business tests focused while a dedicated suite verifies
the guard.
