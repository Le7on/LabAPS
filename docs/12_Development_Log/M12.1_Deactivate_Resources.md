# Development Log

## Milestone M12.1 — Deactivate / Activate Resources

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Complete

**Owner:** Development Team

**Completed:** 2026-07-08

---

# 1. Objective

Let users deactivate (and reactivate) equipment, staff and projects. Deactivated
resources are excluded from scheduling, since the Planning Context snapshot only
captures active resources.

---

# 2. Scope

- Repository `set_active(id, active)` for equipment, staff, project.
- `SetResourceActiveUseCase` (one use case, dispatches by resource kind).
- API: `POST /{equipment|staff|projects}/{id}/{deactivate|activate}`.
- UI: deactivate/activate button on each row in Equipment/Staff/Projects.

---

# 3. Deliverables

| Component | Location                                                      |
| --------- | ------------------------------------------------------------- |
| Repos     | equipment/staff/project repository `set_active`               |
| Use case  | `application/set_resource_active.py`                          |
| APIs      | equipment/staff/project blueprints                            |
| UI        | Equipment/Staff/Projects views + laboratory store `setActive` |
| Tests     | `backend/modules/laboratory/tests/test_deactivate_api.py`     |

---

# 4. Out of Scope

- Full field editing (rename, change capabilities/skills) — deactivate is the
  reversible, safe state change; edits are a later step.
- Cascading effects on already-scheduled versions (snapshots are immutable, so
  past schedules are unaffected by design).

---

# 5. Acceptance Criteria

- Deactivating a resource sets active=false and is reflected in listings.
- Reactivating restores active=true.
- Deactivating an unknown resource returns 404.

# 8. Completion Record

**Date:** 2026-07-08

Deactivation is modelled as a status flag change via one dispatching use case,
keeping the three resource types consistent. Because scheduling reads immutable
Planning Context snapshots, deactivating a resource affects only future
generations, not already-scheduled versions.
