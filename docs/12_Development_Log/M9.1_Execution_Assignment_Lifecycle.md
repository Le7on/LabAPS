# Development Log

## Milestone M9.1 — Execution: Assignment Lifecycle

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Complete

**Owner:** Development Team

**Completed:** 2026-07-08

---

# 1. Objective

Deliver the first Execution slice: transition an Assignment's execution status
(Pending -> Ready -> Running -> Completed | Failed | Cancelled) per the Assignment
State Machine (State Model, chapter 4). This is Phase 5 of the Bootstrap Plan.

---

# 2. Scope

- `AssignmentStatus` enum and transition rules (`next_status`, `require_reason`)
  enforcing the state table; terminal states reject actions.
- Publishing a Plan Version moves its Pending assignments to Ready (BR-AS-002).
- Execution use cases: Start / Complete / Fail / Cancel (Fail and Cancel require
  a reason, BR-AS-005/006).
- API commands: `POST /api/v1/executions/{id}/{start|complete|fail|cancel}`.
- Assignment ORM gains a `reason` column; default status is now `pending`.

---

---

# 4. Out of Scope

- Execution Record / Execution Log history entities.
- Resource occupancy checks and calendar integration.
- Domain events (AssignmentStarted, ...).

---

# 5. Acceptance Criteria

- Publishing a version marks its assignments Ready.
- Ready -> Running -> Completed/Failed works; invalid transitions return 409.
- Fail and Cancel require a reason (422 otherwise).
- Terminal assignments reject further actions.

---

# 6. Verification

- `test_assignment_status.py`: forward transitions, invalid transitions, terminal
  rejection, reason requirement.
- `test_executions_api.py`: publish -> ready; start -> complete; complete before
  start is 409; fail requires reason then succeeds and records it.
- 65 tests pass overall. Ruff clean; migration verified.

---

# 7. Next Milestone

Execution Record / history and resource occupancy; Execution frontend view.

---

# 8. Completion Record

**Date:** 2026-07-08

Execution changes only the assignment's execution status; it never modifies the
owning Plan Version (architectural rule 6). The transition table is data-driven
(`_TRANSITIONS`), so adding states or actions is a table edit. Publishing bridges
planning and execution by moving Pending assignments to Ready in the same
transaction.
