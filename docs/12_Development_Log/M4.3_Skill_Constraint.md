# Development Log

## Milestone M4.3 — Skill Constraint (Multi-Resource Assignment)

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Complete

**Owner:** Development Team

**Completed:** 2026-07-08

---

# 1. Objective

Extend the scheduler to assign Staff (by skill) alongside Equipment (by
capability), so an operation that needs both gets a capable machine and a
qualified operator.

Implements the Capability and Skill/Qualification constraint categories together
(Constraint Framework; Scheduling Model Resource Graph).

---

# 2. Scope

- Generalized the Scheduling Model: resources have a `kind` ("equipment",
  "staff") and a `provides` attribute set; tasks carry `requirements` as
  (kind, value) pairs. `ScheduledTask` holds per-kind `ResourceAssignment`s.
- OR-Tools adapter assigns, per kind present, exactly one eligible resource and
  enforces no-overlap per resource across all kinds. Infeasible when a required
  kind has no eligible resource.
- PlanningProblem: `Resource` gains `kind` + `provides`; `Operation` gains
  `required_skill`.
- Model builder maps required capability -> equipment requirement, required
  skill -> staff requirement.
- Schedule-from-workflow feeds active Equipment (capability) and Staff (skill) as
  resources; assignments expose `equipmentId` and `staffId`.
- FakeSolverAdapter and AssignmentBuilder updated for multi-kind assignment
  (Assignment keeps `resource_id` for compatibility, adds `equipment_id`,
  `staff_id`).

# 4. Out of Scope

- Qualification expiry / calendar / policy constraints.
- Objective changes (still makespan; ADR-007).

---

# 5. Acceptance Criteria

- An operation requiring a capability and a skill is assigned a capable equipment
  and a skilled staff member.
- A resource (of any kind) does one operation at a time.
- Missing a required kind's resource is infeasible.
- Existing single-resource (equipment-only) behaviour is preserved.

---

# 6. Verification

- Real CP-SAT: equipment-only matching; equipment+staff matching; single-resource
  serialization (makespan 7); two-resource parallelism (makespan 4); infeasible.
- Engine (fake): equipment assignment; equipment+staff assignment; infeasible.
- End-to-end API: schedule-from-workflow assigns both equipmentId and staffId.
- 53 tests pass. Ruff clean.

# 8. Completion Record

**Date:** 2026-07-08

The model was generalized from a single equipment dimension to per-kind resource
assignment without breaking existing tests (Assignment still exposes
`resource_id` as the equipment). This keeps the Constraint Framework's Capability
and Skill categories as one uniform mechanism, so adding future resource kinds is
a data change rather than a solver rewrite.
