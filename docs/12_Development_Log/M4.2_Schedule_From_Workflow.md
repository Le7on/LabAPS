# Development Log

## Milestone M4.2 — Schedule From Persisted Laboratory Data

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Complete

**Owner:** Development Team

**Completed:** 2026-07-08

---

# 1. Objective

Connect the Laboratory Definition domain to the Scheduling Engine end to end:
build the PlanningProblem from persisted data (operations from a Workflow
Definition, resources from the active Equipment pool) instead of hand-fed input.

This is the first real Laboratory -> Planning -> Scheduling integration.

---

# 2. Scope

- `GenerateScheduleFromWorkflowUseCase`: loads a Plan Version, a Workflow
  Definition and the active Equipment via the Unit of Work; builds the
  PlanningProblem (operations from the workflow's Operation Definitions,
  resources from equipment capability sets) and runs the injected Scheduling
  Engine.
- API: `POST /api/v1/plans/{planId}/versions/{versionId}/schedule-from-workflow`
  with `{ "workflowDefinitionId": ... }`.

---

# 3. Deliverables

| Component | Location                                                                  |
| --------- | ------------------------------------------------------------------------- |
| Use case  | `backend/modules/planning/application/generate_schedule_from_workflow.py` |
| API       | `backend/modules/planning/api/plans_api.py`                               |
| Tests     | `backend/modules/planning/tests/test_schedule_from_workflow_api.py`       |

---

# 4. Out of Scope

- Persisting the resulting Assignments and creating Workflow/Operation Instances
  (Planning aggregate extension, later).
- Staff as a scheduling resource (Skill constraint — Phase D / next milestone).
- Objective beyond makespan (unchanged from M4.1; ADR-007).

---

# 5. Acceptance Criteria

- Scheduling reads operations from a persisted Workflow Definition and resources
  from persisted Equipment.
- An operation requiring a capability is assigned to a capable equipment from the
  pool.
- Unknown plan/version/workflow return 404; empty workflow returns a validation
  error.

---

# 6. Verification

- `test_schedule_from_workflow_api.py`: end-to-end via real CP-SAT — a pcr
  operation is assigned to the only pcr-capable equipment; unknown workflow 404.
- 43 tests pass overall. Ruff clean.

---

# 8. Completion Record

**Date:** 2026-07-08

The use case reads across modules (planning + laboratory) at the application
layer, which is allowed; the domain and engines remain module-independent.
Operation identity is the Operation Definition id, and workflow dependencies
reference the same ids, so precedence carries through to the scheduler
unchanged.
