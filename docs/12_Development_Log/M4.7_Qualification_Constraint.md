# Development Log

## Milestone M4.7 — Qualification Constraint

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Complete

**Owner:** Development Team

**Completed:** 2026-07-08

---

# 1. Objective

Add the Qualification constraint category: an operation can require a
qualification, and only staff holding a currently-valid (unexpired)
qualification may perform it. Unlike a skill, a qualification can lapse.

This completes the frozen Constraint Framework's core categories (Dependency,
Capability, Skill, Resource, Calendar, Qualification).

---

# 2. Scope

- Staff gains `qualifications` (name -> expiry date | null) with
  `valid_qualifications(reference_date)`.
- Operation Definition / Operation Instance gain `required_qualification`.
- Scheduling model: a task may hold multiple requirements per resource kind;
  `eligible_resources` now requires a resource to provide ALL of them
  (`satisfies_all`); the solver enforces per-kind with `requires(kind)`.
- Model builder maps a required qualification as a STAFF requirement.
- Planning Context snapshot captures staff qualifications; `schedule-instances`
  evaluates validity at today's date and folds valid qualifications into the
  staff member's provided set (expired ones are simply absent -> ineligible).
- Migration adds the three qualification columns.

# 4. Out of Scope

- Qualification as a first-class entity/catalogue (stored as a name->expiry map
  on staff for now).
- Warning when a qualification will expire during the planning horizon.
- Time-phased qualification validity within a schedule (validity is evaluated at
  a single reference date).

---

# 5. Acceptance Criteria

- An operation requiring a qualification is assigned only to staff whose
  qualification is valid (and who also have any required skill).
- An expired qualification makes the operation infeasible.
- Existing skill/capability/calendar behaviour is unchanged.

# 8. Completion Record

**Date:** 2026-07-08

Qualification reuses the multi-requirement mechanism rather than a bespoke path:
a staff resource "provides" its skills plus its valid qualifications, and a task
requiring both must find a resource providing both. Validity is resolved when
building the problem from the snapshot, so an expired qualification cleanly drops
the resource from eligibility. This keeps the solver uniform across constraint
categories.
