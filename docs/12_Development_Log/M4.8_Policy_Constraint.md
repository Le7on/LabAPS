# Development Log

## Milestone M4.8 — Policy Constraint (Frozen Planning Window)

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Complete

**Owner:** Development Team

**Completed:** 2026-07-08

---

# 1. Objective

Add the Policy constraint category via the frozen planning window: no operation
may start before a frozen boundary (near-term work is locked). This completes all
Constraint Framework categories.

Note: two other documented policies are already enforced elsewhere — published
plans are immutable (M3.2) and the planning horizon bounds scheduling (M4.1).

---

# 2. Scope

- `SchedulingModel.frozen_until` and `PlanningPolicies.frozen_until`; the model
  builder threads it through.
- OR-Tools adapter sets each task's earliest start to the frozen boundary.
- `schedule-instances` accepts an optional `frozenUntil` in the request body.

---

---

# 4. Out of Scope

- Periodic policies (e.g. "FV required every 14 days") — need recurrence modelling.
- SOP-driven policy catalogue; per-resource frozen windows.

---

# 5. Acceptance Criteria

- With no frozen window, scheduling starts at 0.
- A frozen boundary pushes all starts to at or after it.
- A frozen window leaving no room before the horizon is infeasible.

# 8. Completion Record

**Date:** 2026-07-08

The frozen window is enforced as the earliest-start lower bound on every task's
start variable, so it composes cleanly with dependency, resource, calendar and
qualification constraints. With this, all Constraint Framework categories
(Dependency, Capability, Skill, Resource, Calendar, Qualification, Policy) have a
working implementation.
