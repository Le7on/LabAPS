# Development Log

## Milestone M5.2 — Staff and Workflow Definition

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Complete

**Owner:** Development Team

**Completed:** 2026-07-07

---

# 1. Objective

Extend the Laboratory Definition domain with Staff (operators with skills) and
Workflow Definition (workflow templates owning Operation Definitions). These are
the real resource and operation sources the Scheduling Engine will consume.

Implements ADR-003 (Workflow Definition vs Workflow Instance), ADR-004 (Operation
Definition vs Operation Instance) and the Business Object Model (Staff owns
Skills; Operations require Capability and Skill).

---

# 2. Scope

Staff:

- Domain entity `Staff` (staff code, name, skill set, active) with `has_skill`
  and `deactivate` behaviour.
- ORM, repository, DTO, create/list use cases and `/api/v1/staff` endpoints.

Workflow Definition:

- Aggregate root `WorkflowDefinition` owning `OperationDefinition` entities;
  `add_operation()` is aggregate-owned.
- `OperationDefinition` carries operation type, duration, required capability,
  required skill and dependencies (no concrete resources — those are Assignment).
- ORM (one-to-many cascade within the aggregate), repository, DTO, create/list
  use cases and `/api/v1/workflow-definitions` endpoints.

Infrastructure:

- Repositories added to the Unit of Work (`uow.staff`,
  `uow.workflow_definitions`).
- ORM registered; blueprints registered in the application factory.
- Alembic migration for equipment, staff, workflow_definition and
  operation_definition.
