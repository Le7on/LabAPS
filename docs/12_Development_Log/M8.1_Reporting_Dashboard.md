# Development Log

## Milestone M8.1 — Reporting Dashboard

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Complete

**Owner:** Development Team

**Completed:** 2026-07-08

---

# 1. Objective

Deliver the first slice of the Reporting module: a read-only dashboard summary of
aggregate counts across the system.

Establishes the read-only Query Service pattern (lightweight CQRS): reads query
the ORM directly, avoiding the domain-mapping cost that only writes need.

---

# 2. Scope

- `DashboardQueryService`: read model computing counts (plans, plan versions,
  published versions, equipment, active equipment, staff, workflow definitions)
  by querying ORM tables directly.
- `GetDashboardUseCase`: read-only use case (session, no Unit of Work / commit).
- API: `GET /api/v1/reports/dashboard`.

---

# 3. Deliverables

| Component     | Location                                                 |
| ------------- | -------------------------------------------------------- |
| Query service | `backend/modules/reporting/query/dashboard_query.py`     |
| Use case      | `backend/modules/reporting/application/get_dashboard.py` |
| API           | `backend/modules/reporting/api/dashboard_api.py`         |
| Tests         | `backend/modules/reporting/tests/test_dashboard_api.py`  |

---

# 4. Out of Scope

- KPI, equipment utilization, material forecast and export reports.
- A reporting frontend view.

---

# 5. Acceptance Criteria

- The dashboard returns counts for the main entities.
- Counts reflect created entities.
- Reporting performs no writes.

---

# 6. Verification

- `test_dashboard_api.py`: empty dashboard; counts reflect created plan,
  equipment, staff and workflow definition.
- 55 tests pass overall. Ruff clean.

---

# 7. Next Milestone

Add KPI and utilization reports, and a reporting frontend view.

---

# 8. Completion Record

**Date:** 2026-07-08

This applies a design recommendation: Reporting is read-only, so it uses a Query
Service reading the ORM directly instead of the write-oriented repository +
domain mapping. The read model returns camelCase counts consistent with the API
envelope contract (ADR-012).
