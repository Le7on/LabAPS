# Development Log

## Milestone M4.5 — Calendar Constraint (Resource Availability Windows)

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Complete

**Owner:** Development Team

**Completed:** 2026-07-08

---

# 1. Objective

Add the Calendar constraint category to the scheduler: a resource is available
only within defined time windows, and a task assigned to it must fit entirely
within one window.

---

# 2. Scope

- `SchedulingResource.windows` (list of `[start, end)`); empty means always
  available. Threaded from the PlanningProblem `Resource` through the model
  builder.
- OR-Tools adapter: for a windowed resource, when a task is assigned to it, the
  task interval must fall within exactly one window (enforced only when the
  assignment literal is true).
- Infeasible when no window can fit a required task.

---

# 3. Deliverables

| Component        | Location                                                 |
| ---------------- | -------------------------------------------------------- |
| Model            | `backend/engines/scheduling/scheduling_model.py`         |
| Builder          | `backend/engines/scheduling/scheduling_model_builder.py` |
| Planning problem | `backend/engines/planning/planning_problem.py`           |
| Solver           | `backend/solver/adapter/ortools_solver_adapter.py`       |
| Tests            | `backend/solver/tests/test_ortools_calendar.py`          |

---

# 4. Out of Scope

- Deriving windows from persisted shift/holiday/leave calendars (the Planning
  Context snapshot does not yet carry calendar windows; windows are supported by
  the engine and ready to be populated).
- Setup/teardown or changeover times.

---

# 5. Acceptance Criteria

- A task assigned to a windowed resource is scheduled within one window.
- No fitting window makes the problem infeasible.
- A later window is used when earlier ones are too short.

---

# 6. Verification

- `test_ortools_calendar.py` (real CP-SAT): within-window placement;
  infeasible-when-no-fit; later-window selection.
- 84 tests pass overall. Ruff clean. No schema change.

---

# 7. Next Milestone

Populate resource windows from persisted shift/calendar data into the Planning
Context snapshot, so scheduling honors real availability.

---

# 8. Completion Record

**Date:** 2026-07-08

The Calendar constraint is enforced per assignment: the window fit is only
required when the task is actually assigned to that resource, so it composes with
capability/skill matching and no-overlap. Windows are an engine-level capability
now; wiring real calendar data into the snapshot is the next step.
