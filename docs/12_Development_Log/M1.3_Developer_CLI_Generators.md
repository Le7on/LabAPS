# Development Log

## Milestone M1.3 — Developer CLI Code Generators

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Complete

**Owner:** Development Team

**Completed:** 2026-07-07

---

# 1. Objective

Complete the developer scaffold by adding code generation to the CLI, so new
modules, domain entities and use cases are created from a single command that
follows the documented implementation templates.

Implements the CLI roadmap from M1.1 (`new module`, `new usecase`) and the
Engineering Baseline (the CLI is the only supported project generation
mechanism; manual scaffolding is discouraged).

---

# 2. Scope

- `labaps.py new module <name>` — generates a full Module-First skeleton
  (api / application / domain / repository / dto / tests, plus domain
  aggregates / entities / enums packages) with a starter entity, repository,
  API blueprint and test.
- `labaps.py new entity <module> <name>` — generates a domain entity following
  the Domain Entity Template (behaviour-first, invariant-protecting,
  framework-free).
- `labaps.py new usecase <module> <name>` — generates a use case following the
  Use Case Template (one action, one transaction via the Unit of Work).
- Name conversion helpers (snake / Pascal / camel / title).
- Generation never overwrites existing files (safe to re-run).

---

# 3. Deliverables

| Component          | Location                              |
| ------------------ | ------------------------------------- |
| Naming helpers     | `tools/naming.py`                     |
| Generator          | `tools/generator.py`                  |
| CLI wiring         | `tools/labaps.py` (`new` subcommands) |
| Code-gen templates | `tools/templates/codegen/`            |
| Tests              | `tools/tests/`                        |

---

# 4. Out of Scope

- Generating ORM models, DTOs and migrations for new modules (the module
  skeleton includes a repository stub; ORM/migration wiring is done per module).
- Automatically registering new blueprints in the application factory (the CLI
  prints a reminder instead).

---

# 5. Acceptance Criteria

- `new module` creates the full layered package with importable, lint-clean
  starter code and a passing generated test.
- `new entity` and `new usecase` render correct class names from a human name.
- Generation does not overwrite existing files.

---

# 6. Verification

- `tools/tests/test_naming.py`: case conversions and context (parametrized).
- `tools/tests/test_generator.py`: module skeleton, entity and use case
  generation into a temp root with name-substitution assertions.
- 9 tools tests pass. Manual smoke: generated a sample module — it imported,
  passed `ruff check`, and its generated test passed; then removed.
- Backend suite unaffected (24 tests still pass).

---

# 7. Next Milestone

Extend generators as needed (ORM model + migration scaffolding, DTO templates)
