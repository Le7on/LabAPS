# Development Log

## Milestone M3.1 — Planning Domain (Plan Slice)

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Complete

**Owner:** Development Team

**Completed:** 2026-07-07

---

# 1. Objective

Implement the first vertical slice of the Planning core domain: the Plan
aggregate root and its Plan Versions, from pure-Python domain through repository,
use cases and REST API.

Implements Phase 3 of the Bootstrap Plan and ADR-001 (Plan as Aggregate Root),
ADR-002 (Plan + Plan Version).

---

# 2. Scope

- Domain (framework-free): `PlanStatus` / `VersionType` / `PlanVersionStatus`
  enums, `PlanVersion` entity, `Plan` aggregate root. `plan_code` is derived as
  `PLAN-<planning_horizon>`.
- Repository converting ORM <-> domain (one repository per aggregate root).
- Use cases: create plan, list plans, get plan, create plan version, generate
  schedule (wiring to the Scheduling Engine — see M4.1).
- REST API: `POST/GET /api/v1/plans`, `GET /api/v1/plans/{id}`,
  `POST /api/v1/plans/{id}/versions`,
  `POST /api/v1/plans/{id}/versions/{versionId}/schedule`.

# 4. Out of Scope

- Plan Version approve/publish lifecycle transitions.
- Building the Planning Problem from persisted Workflow/Operation Instances and
  the Planning Context snapshot (a later milestone; operations are currently
  supplied in the schedule request).

---

# 5. Acceptance Criteria

- Creating a plan derives `planCode` from the planning horizon.
- Plans can be created, listed and retrieved through the API.
- A plan version can be created on an existing plan.
- Planning domain objects pass unit tests.
