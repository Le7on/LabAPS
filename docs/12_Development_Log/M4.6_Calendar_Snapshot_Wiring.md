# Development Log

## Milestone M4.6 — Calendar Snapshot Wiring

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Complete

**Owner:** Development Team

**Completed:** 2026-07-08

---

# 1. Objective

Feed real resource availability into the calendar constraint (M4.5): Equipment
and Staff carry availability windows, captured in the immutable Planning Context
snapshot and applied as resource windows during instance-based scheduling.

---

# 2. Scope

- `Equipment.availability` and `Staff.availability` (windows `[start, end)`),
  validated (end > start); ORM JSON columns; repository/DTO/use case/API carry
  them.
- `GenerateWorkflowInstance` snapshots each active resource's availability into
  the Planning Context.
- `ScheduleInstances` builds resources with `windows` from the snapshot, so the
  solver's Calendar constraint uses real data.
- Alembic migration adds the two availability columns.

---

# 3. Deliverables

| Component  | Location                                                         |
| ---------- | ---------------------------------------------------------------- |
| Entities   | `equipment.py`, `staff.py` (availability + validation)           |
| ORM        | `equipment_orm.py`, `staff_orm.py` (availability JSON)           |
| DTO/UC     | equipment/staff DTOs + create use cases carry availability       |
| Snapshot   | `application/generate_workflow_instance.py`                      |
| Scheduling | `application/schedule_instances.py` (windows from snapshot)      |
| Migration  | `migrations/versions/*add_availability_to_equipment_and_staff*`  |
| Tests      | `backend/modules/planning/tests/test_calendar_scheduling_api.py` |

---

# 4. Out of Scope

- Shift/holiday/leave calendars as first-class entities (availability is entered
  directly on equipment/staff for now).
- Recurring windows and time-of-day calendars (windows are integer horizon
  units, consistent with the current scheduler).

---

# 5. Acceptance Criteria

- Equipment/Staff can be created with availability windows.
- Windows are captured in the Planning Context snapshot at generation time.
- Instance scheduling places operations within a resource's available window.

---

# 6. Verification

- `test_calendar_scheduling_api.py`: equipment available only in [10, 20) ->
  the scheduled operation lands inside that window.
- 85 tests pass overall. Alembic migration verified. Ruff clean.

---

# 7. Next Milestone

Shift/holiday calendars as reusable entities that populate resource availability;
time-of-day calendars.

---

# 8. Completion Record

**Date:** 2026-07-08

This completes the calendar path started in M4.5: the solver capability now runs
on real, snapshot-captured availability (ADR-008), so a scheduled version stays
reproducible even if a resource's availability later changes. Availability is
entered directly on equipment/staff; a dedicated shift/calendar model is a future
step.
