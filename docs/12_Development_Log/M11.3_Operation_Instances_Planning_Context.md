# Development Log

## Milestone M11.3 — Operation Instances & Planning Context

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Complete

**Owner:** Development Team

**Completed:** 2026-07-08

---

# 1. Objective

Deepen the core chain: materialize a Workflow Definition into version-owned
Workflow/Operation Instances and capture an immutable Planning Context snapshot,
then schedule against the snapshot rather than live configuration (ADR-003,
ADR-004, ADR-008).

---

# 2. Scope

- `WorkflowInstanceORM` + `OperationInstanceORM` (Plan Version aggregate) and
  `PlanningContextORM` (immutable snapshot: equipment/staff/solver profile).
- `WorkflowInstanceRepository`: generate instances + snapshot for a version
  (replace on regenerate); list operation instances; read the context.
- `GenerateWorkflowInstanceUseCase`: definition -> instances + snapshot of the
  active resources at generation time.
- `ScheduleInstancesUseCase`: build the PlanningProblem from persisted operation
  instances and the snapshot (not live resources), schedule, persist assignments
  (referencing operation instance ids), mark the version Scheduled.
- API: `POST /plans/{id}/versions/{vid}/generate-instances` and
  `POST /plans/{id}/versions/{vid}/schedule-instances`.
- Alembic migration for the three tables.

---

# 4. Out of Scope

- Project aggregate and per-project demand attribution.
- Calendar/shift snapshots in the context (equipment/staff/solver profile only
  for now).
- Operation instance status transitions beyond scheduling.

---

# 5. Acceptance Criteria

- Generating instances creates one operation instance per definition operation
  plus a Planning Context snapshot.
- Scheduling instances uses the snapshot resources and persists assignments
  referencing operation instance ids.
- Scheduling without prior generation returns 422.

# 8. Completion Record

**Date:** 2026-07-08

This realizes ADR-008: scheduling reads the version's immutable snapshot, so a
scheduled version is reproducible regardless of later changes to live equipment
or staff. The earlier `schedule-from-workflow` path (live resources, transient
operations) remains for quick scheduling; `schedule-instances` is the durable,
snapshot-based path. Assignments now reference operation instance ids.
