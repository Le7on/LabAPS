# State Model

## Chapter 4 - Assignment State Machine

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Architecture Baseline

---

# 1. Purpose

This document defines the lifecycle of an Assignment.

An Assignment represents one scheduled execution of an Operation Instance on a specific Equipment, by a specific Staff member, during a specific Shift.

Unlike a Plan Version, an Assignment represents executable work.

Its lifecycle reflects execution rather than planning.

---

# 2. Design Philosophy

Planning creates Assignments.

Execution changes Assignment state.

Scheduling never modifies an Assignment after the associated Plan Version has been Published.

Execution updates only the execution status.

---

# 3. Lifecycle

The Assignment lifecycle is intentionally simple.

```text
Pending

↓

Ready

↓

Running

↓

Completed

      ↘

      Failed

      ↘

    Cancelled
```

Each state has one clear operational meaning.

---

# 4. Pending

## Meaning

The Assignment has been generated by the Scheduler.

The owning Plan Version has not yet been Published.

The Assignment is not executable.

---

## Allowed Actions

- View

---

## Forbidden Actions

- Start
- Complete

---

# 5. Ready

## Meaning

The Plan Version has been Published.

The Assignment is ready for execution.

Resources have been reserved.

---

## Allowed Actions

- Start
- Cancel (Business Policy Dependent)

---

# 6. Running

## Meaning

Execution has started.

The assigned Equipment and Staff are occupied.

---

## Allowed Actions

- Complete
- Fail

---

# 7. Completed

## Meaning

Execution finished successfully.

No further execution actions are allowed.

The Assignment becomes read-only.

---

# 8. Failed

## Meaning

Execution could not be completed successfully.

Examples

- Instrument Error
- QC Failure
- Operator Aborted
- Unexpected Exception

Failure reason shall be recorded.

---

# 9. Cancelled

## Meaning

The Assignment will not be executed.

Typical reasons

- Plan superseded before execution
- Equipment unavailable before execution
- Manual cancellation

Cancellation reason shall be recorded.

---

# 10. State Transition Diagram

```text
Pending

↓

Ready

↓

Running

├────────────► Completed

├────────────► Failed

└────────────► Cancelled
```

Backward transitions are prohibited.

---

# 11. Transition Rules

| Current | Action               | Next      |
| ------- | -------------------- | --------- |
| Pending | Publish Plan Version | Ready     |
| Ready   | Start                | Running   |
| Ready   | Cancel               | Cancelled |
| Running | Complete             | Completed |
| Running | Fail                 | Failed    |

All other transitions are invalid.

---

# 12. Business Rules

BR-AS-001

Assignments are created in the Pending state.

---

BR-AS-002

Publishing a Plan Version changes all Pending Assignments to Ready.

---

BR-AS-003

Only Ready Assignments may start execution.

---

BR-AS-004

Completed Assignments are immutable.

---

BR-AS-005

Failed Assignments require a failure reason.

---

BR-AS-006

Cancelled Assignments require a cancellation reason.

---

# 14. API Behaviour

Examples

Start Assignment

```http
POST /api/v1/executions/{assignmentId}:start
```

Complete Assignment

```http
POST /api/v1/executions/{assignmentId}:complete
```

Fail Assignment

```http
POST /api/v1/executions/{assignmentId}:fail
```

Cancel Assignment

```http
POST /api/v1/executions/{assignmentId}:cancel
```

Invalid transitions return

```text
409 Conflict
```

with an appropriate business error code.

---

# 15. Domain Behaviour

Assignment exposes explicit business methods.

```text
mark_ready()

start()

complete()

fail(reason)

cancel(reason)
```

The execution status shall never be modified by directly assigning the status field.

---

# 16. Domain Events

State transitions generate business events.

Examples

- AssignmentReady
- AssignmentStarted
- AssignmentCompleted
- AssignmentFailed
- AssignmentCancelled

These events support:

- Audit
- Dashboard updates
- Notifications
- Future integrations

---

# 17. Architectural Rules

1. Assignment state represents execution only.

2. Scheduling creates Assignments but does not manage execution.

3. Execution changes Assignment state.

4. Completed, Failed and Cancelled Assignments are terminal states.

5. Every state transition shall be auditable.

6. Assignment state changes never modify the owning Plan Version state directly.
