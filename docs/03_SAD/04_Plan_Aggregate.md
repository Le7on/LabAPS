# Software Architecture Design

## Chapter 4 - Plan Aggregate Design

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Draft

---

# 1. Purpose

This document defines the Plan Aggregate, which is the core aggregate of the Planning Domain.

Every planning activity in Lab APS starts from a Plan and produces data owned by that Plan.

The Plan Aggregate defines:

* Aggregate Boundary
* Aggregate Ownership
* Lifecycle
* Invariants
* State Transitions

All planning-related business objects belong to exactly one Plan.

---

# 2. Why Plan is the Aggregate Root

Lab APS is not scheduling individual Operations.

Lab APS schedules an entire production period.

Examples

```text
Week 32 Production Plan

Week 33 Production Plan

August Production Plan
```

Everything generated during planning belongs to one Plan.

Therefore:

**Plan is the Aggregate Root.**

---

# 3. Aggregate Structure

```text
Plan
│
├── Planning Context
│
├── Demands
│
├── Workflow Instances
│      │
│      └── Operations
│
├── Assignments
│
├── Material Forecast
│
├── KPI
│
└── Versions
```

No object may exist without a parent Plan.

---

# 4. Plan Identity

Each Plan is uniquely identified.

Suggested business identifier:

```text
PLAN-2026-W32-V1
```

Meaning

| Part | Description      |
| ---- | ---------------- |
| PLAN | Object Type      |
| 2026 | Planning Year    |
| W32  | Planning Horizon |
| V1   | Plan Version     |

The database may additionally use an internal UUID.

---

# 5. Aggregate Responsibilities

Plan is responsible for:

* Managing Demand
* Managing Workflow Instances
* Managing Assignments
* Managing Planning Context
* Managing Plan Status
* Publishing
* Version Control
* Forecast Ownership

Plan is **not** responsible for:

* Equipment Definition
* Staff Definition
* Workflow Template Definition
* Scheduling Algorithm

---

# 6. Aggregate Invariants

The following rules must always be true.

## INV-001

A Plan always has exactly one Planning Context.

---

## INV-002

A Demand always belongs to one Plan.

---

## INV-003

A Workflow Instance always belongs to one Plan.

---

## INV-004

An Assignment always belongs to one Plan.

---

## INV-005

Material Forecast belongs to one Plan.

---

## INV-006

Published Plans are immutable.

---

## INV-007

Only one Published Version may exist for the same Planning Horizon.

---

# 7. Planning Context

Planning Context captures the environment used to generate the schedule.

It contains immutable snapshots.

```text
Planning Context

├── Calendar Snapshot

├── Staff Snapshot

├── Equipment Snapshot

├── Shift Profile

├── Solver Profile

└── Planning Parameters
```

The Planning Context guarantees reproducibility.

---

# 8. Demand

Demand defines **what** must be completed.

Demand never defines:

* Equipment
* Staff
* Shift

Demand belongs entirely to the Plan.

---

# 9. Workflow Instance

Workflow Instances are generated automatically.

The user never creates Workflow Instances manually.

Example

```text
Demand

PNG × 3

↓

Generate

PNG001

PNG002

PNG003
```

---

# 10. Operations

Operations belong to Workflow Instances.

Operations are generated.

Operations are immutable after scheduling.

Examples

```text
PNG001

SMDP

↓

SAP
```

Operations describe work.

Assignments describe execution.

---

# 11. Assignments

Assignments are generated by the Scheduling Engine.

Assignment binds:

```text
Operation

↓

Equipment

↓

Staff

↓

Shift
```

Assignments may change while the Plan is in Draft or Reviewed state.

Assignments cannot change after Publish.

---

# 12. Material Forecast

Material Forecast is generated from Assignments.

It is a derived object.

It is never edited manually.

Material Forecast contains:

* Daily Consumption
* Weekly Consumption
* Inventory Warnings

---

# 13. KPI

KPIs are generated automatically.

Examples

* Equipment Utilization
* Staff Utilization
* Planning Completion
* Solver Runtime
* Demand Completion Rate

KPIs are read-only.

---

# 14. Plan Lifecycle

```text
Draft
    │
    ▼
Validated
    │
    ▼
Scheduled
    │
    ▼
Reviewed
    │
    ▼
Published
    │
    ▼
Executing
    │
    ▼
Completed
    │
    ▼
Archived
```

The lifecycle is strictly ordered.

Backward transitions are not allowed.

# 16. Version Strategy

A Plan Version represents one complete planning result.

Example

```text
Week 32

V1 Initial Plan

↓

V2 Equipment Maintenance

↓

V3 Demand Update
```

Each version has its own:

* Planning Context
* Schedule
* Forecast
* KPI

Only one version may be Published.

---

# 17. Aggregate Boundary

Objects inside the Plan Aggregate:

```text
Plan

Planning Context

Demand

Workflow Instance

Operation

Assignment

Material Forecast

KPI
```

Objects outside the Plan Aggregate:

```text
Staff

Equipment

Workflow Template

Shift Definition

Capability

Skill
```

Outside objects are referenced through snapshots in the Planning Context.

---

# 18. Architectural Implications

The Plan Aggregate defines the structure of the Planning Domain.

Future implementations shall follow these principles:

* Every scheduling request operates on one Plan.
* The Scheduling Engine receives a Plan as input.
* The Scheduling Engine returns Assignments to the same Plan.
* Material Forecast and KPI are regenerated whenever the schedule changes.
* No object outside the Plan Aggregate may directly modify Plan-owned objects.
