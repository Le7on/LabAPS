# Software Requirements Specification

## Chapter 5 - Business Object Model

**Project:** Laboratory Advanced Planning & Scheduling Platform (Lab APS)

**Version:** 1.0

**Status:** Draft

---

# 1. Purpose

This document defines the core business objects of Lab APS.

Business Objects represent the fundamental concepts of the planning domain.

These objects are independent of:

- Database Design
- REST API
- User Interface
- OR-Tools
- Programming Language

Every subsequent design artifact shall derive from this model.

This document is the **Single Source of Truth (SSOT)** for the domain model.

---

# 2. Domain Overview

Lab APS revolves around one central business object:

```text
Plan
```

Everything else exists because of a Plan.

The overall domain relationship is shown below.

```text
Plan
│
├── Planning Context
├── Demand
├── Workflow Instance
├── Assignment
├── Material Forecast
└── KPI
```

Plan is the Aggregate Root of the domain.

---

# 3. Plan

## Description

A Plan represents one laboratory production planning cycle.

Examples

```text
Week 32 Production Plan
```

```text
August Planning
```

A Plan owns all business objects created during planning.

---

## Lifecycle

```text
Draft

↓

Ready

↓

Scheduled

↓

Reviewed

↓

Published

↓

Executing

↓

Completed

↓

Archived
```

---

## Responsibilities

A Plan is responsible for

- Planning Horizon
- Demand
- Workflow Generation
- Scheduling Result
- Material Forecast
- Planning KPI

A Plan is NOT responsible for

- Equipment Configuration
- Workflow Definition
- User Management

---

# 4. Planning Context

## Description

Planning Context represents the planning environment used to generate a Plan.

It captures the state of the laboratory at planning time.

Planning Context ensures that historical Plans remain reproducible.

---

## Contains

Calendar

Staff Availability

Equipment Availability

Shift Definition

Solver Parameters

Planning Strategy

---

## Example

```text
Planning Context

Week 32

Holiday Calendar A

Shift Template V1

Equipment Snapshot

Staff Snapshot
```

Planning Context is immutable after scheduling.

---

# 5. Demand

## Description

Demand represents laboratory production requirements.

Demand belongs to exactly one Plan.

---

## Contains

Project

Required Quantity

Priority

(Optional)

---

## Example

| Project | Quantity |
| ------- | -------: |
| FV      |        6 |
| PNG     |       18 |
| OPA     |       12 |

Demand does not specify resources.

---

# 6. Workflow Template

## Description

Workflow Template defines the standard execution process for a Project.

Workflow Templates belong to Master Data.

Workflow Templates are reusable.

Workflow Templates are never scheduled directly.

---

## Example

384 PNG

```text
SMDP

↓

Output Plate

↓

SAP
```

---

Workflow Template defines

Operations

Dependencies

Intermediate Outputs

Material BOM

---

# 7. Workflow Instance

## Description

Workflow Instance is generated automatically from Demand.

Workflow Instance belongs to exactly one Plan.

---

## Example

Demand

```text
PNG

Quantity

3
```

Generates

```text
PNG001

PNG002

PNG003
```

---

Workflow Instance owns Operations.

Workflow Instance has execution status.

---

# 8. Operation

## Description

Operation is the smallest executable scheduling unit.

Operations are generated by Workflow Engine.

Operations are allocated by Scheduling Engine.

---

## Example

PNG001

```text
SMDP
```

PNG001

```text
SAP
```

---

Operation contains

Operation Type

Duration

Required Capability

Required Skill

Dependencies

Material BOM Reference

---

Operation never stores

Equipment

Staff

Shift

These belong to Assignment.

---

Status

Equipment belongs to Master Data.

Equipment is independent of any Plan.

Capability

Status

Qualification Status

Maintenance Status

---

Schedule

Workflow

# 11. Staff

Staff belongs to Master Data.

Staff represents laboratory operators.

---

Staff contains

Name

Skills

Status

Working Calendar

Leave

---

Workflow

Project

Assignment History

---

# 12. Capability

Capability describes laboratory hardware features.

Examples

96 Head

384 Head

16 Channel

14 Channel

iSWAP

Operations require Capabilities.

Equipment provides Capabilities.

Scheduling matches the two.

---

# 13. Skill

Skill represents laboratory operator qualifications.

Examples

FV

OPA

PNG

AZ RSV

DiLA

Operations require Skills.

Staff owns Skills.

Scheduling matches the two.

---

# 14. Intermediate Resource

Intermediate Resources are generated during Workflow execution.

Unlike Materials,

Intermediate Resources participate in scheduling.

---

Example

```text
SMDP

↓

Output Plate

↓

SAP
```

Intermediate Resource contains

Creation Time

Expiration Time

Stamp Out Limit

Current Usage

---

# 15. Material

Materials are consumed during laboratory execution.

Materials do not participate in scheduling optimization.

---

Examples

Tips

Plate

Deepwell Block

Seal

FV Kit

---

Materials are maintained in Master Data.

Material usage is calculated after scheduling.

---

# 16. Material Forecast

Material Forecast belongs to one Plan.

Material Forecast is generated after scheduling.

---

Contains

Daily Consumption

Weekly Consumption

Material Summary

Inventory Warning

---

Material Forecast is read-only.

---

# 17. KPI

# 19. Aggregate Rules

The following Aggregate Roots exist in Lab APS.

| Aggregate   | Root              |
| ----------- | ----------------- |
| Planning    | Plan              |
| Workflow    | Workflow Template |
| Master Data | Equipment         |
| Master Data | Staff             |

No business object may modify another Aggregate directly.

All communication shall occur through Application Services.

---

# 20. Architectural Constraints

The following rules shall always be respected.

1.

Plan is the central business object.

2.

Workflow Template belongs to Master Data.

3.

Workflow Instance belongs to a Plan.

4.

Operations are generated.

They are never manually created.

5.

Assignments are scheduling results.

6.

Materials are forecast objects.

They are not scheduling resources.

7.

Business Objects shall remain independent of OR-Tools.

8.

Database tables, REST APIs and Python classes shall all derive from this document.
