# Lab APS — 用户手册

**产品:** 实验室高级计划与排程平台(Lab APS)

**版本:** 1.0

**读者对象:** 实验室计划员、生产负责人(Production LM)与管理员。

**English version:** [User_Manual_EN.md](User_Manual_EN.md)

---

# 1. Lab APS 能做什么

Lab APS 把实验室的资源和工作流程转化为优化后的生产排程,并跟踪其执行。

你需要描述:

- **资源** —— 设备(带能力)和人员(带技能与资质),各自可设可用时间窗。
- **工作流** —— 可复用的工序模板,每道工序有时长,以及所需的能力 / 技能 /
  资质。
- **需求** —— 每个项目需要多少产量,以及紧急程度。

Lab APS 随后生成排程:把每道工序分配给有能力的设备和有资质的操作人员,尊重
前置依赖与可用性,并优先安排紧急需求。你审核、发布,然后驱动执行(开始 / 完成 /
失败 / 取消),全过程留有审计记录。

---

# 2. 关键概念

| 术语                            | 含义                                                 |
| ------------------------------- | ---------------------------------------------------- |
| 设备(Equipment)                 | 一台机器,拥有一组能力(如 "pcr"),可选可用时间窗。     |
| 人员(Staff)                     | 操作人员,拥有技能和资质(资质可过期),可选可用时间窗。 |
| 项目(Project)                   | 提出需求的对象。                                     |
| 工作流定义(Workflow Definition) | 可复用模板:一组带要求的有序工序。                    |
| 工序(Operation)                 | 一个步骤:时长加所需条件(能力、技能、资质)。          |
| 计划(Plan)                      | 面向某个计划周期(如一周)的生产计划。                 |
| 计划版本(Plan Version)          | 对某计划的一次求解尝试,有生命周期(见 §7)。           |
| 需求(Demand)                    | 某项目的请求数量,带优先级(低 / 普通 / 高)。          |
| 计划上下文(Planning Context)    | 生成版本工序时对资源拍下的不可变快照,使排程可复现。  |
| 分配(Assignment)                | 排程结果:某工序在时间轴上落到某设备(与人员)上。      |

# 3. 快速开始

## 3.1 运行应用

桌面模式(单用户推荐):

```text
cd frontend && npm run build
python desktop.py
```

会打开标题为 "Lab APS" 的窗口。若未安装 PyWebView,请在浏览器打开
`http://127.0.0.1:5000`。

开发模式(两个进程):

```text
python run.py                 # 后端 API,http://127.0.0.1:5000
cd frontend && npm run dev     # 前端界面,http://127.0.0.1:5173
```

## 3.2 登录

若启用了认证,应用会显示 **登录** 界面。粘贴你的 API 令牌(token)并登录。你的
名字与角色显示在左侧边栏底部,点 **退出登录** 结束会话。若没有令牌,请向管理员
申请。

# 4. 工作区

左侧边栏用于导航:

- **Dashboard(仪表盘)** —— 全局数量一览。
- **Plans(计划)** —— 创建与查看生产计划。
- **Scheduling(排程)** —— 主工作区:构建、排程并执行计划版本。
- **Laboratory(实验室)**
  - **Projects(项目)** —— 提出需求的项目。
  - **Equipment(设备)** —— 机器及其能力 / 可用性。
  - **Staff(人员)** —— 操作人员、技能、资质与可用性。
  - **Workflows(工作流)** —— 可复用的工序模板。

# 5. 配置实验室数据

一次性配置(变化时再更新)。

## 5.1 项目

Projects → 填写 **项目编码** 和 **名称** → **Add project**。用所在行的按钮停用
项目;停用后的项目不再进入新的排程,但已有排程不受影响。

## 5.2 设备

Equipment → 填写:

- **Code(编码)** 与 **Name(名称)**。
- **Capabilities(能力)** —— 逗号分隔,如 `pcr, spin`。要求某能力的工序只能在
  拥有该能力的设备上运行。
- **Availability(可用性)** —— 可选时间窗,写成 `起-止` 对,如 `0-40, 60-100`。
  留空表示随时可用。(时间使用计划周期的单位。)

## 5.3 人员

Staff → 填写:

- **Code** 与 **Name**。
- **Skills(技能)** —— 逗号分隔,如 `pcr, elisa`。
- **Qualifications(资质)** —— `名称:到期日` 对,如 `gmp:2099-12-31, iso`。只写
  名称(`iso`)表示永不过期。要求某资质的工序只会分配给资质仍然有效的人员。
- **Availability(可用性)** —— 同设备。

## 5.4 工作流

Workflows → 设置 **工作流编码** 与 **名称**,然后添加工序行。每道工序:

- **Operation type(工序类型)** 与 **Duration(时长)**。
- 可选 **Capability(设备能力)**、**Skill(技能)** 与 **Qualification(资质)**。

用 **+ Operation** 增加行、**✕** 删除行,最后 **Create workflow**。

# 6. 计划与排程

进入 **Scheduling**。工作区分四步引导你完成。

## 第 1 步 —— 计划与工作流

选择一个 **Plan**(先在 Plans 里创建)和一个 **Workflow**,点
**Create version & generate**。这会创建一个新的计划版本并生成其工序实例,同时对
当前处于启用状态的设备与人员拍下不可变快照。

## 第 2 步 —— 需求(可选)

选择 **Project**、**Quantity(数量)** 与 **Priority(优先级)**,点
**Add demand**。有需求时,排程器会更倾向于让高优先级的工作更早完成;没有需求时,
它只最小化整体完工时间(makespan)。

## 第 3 步 —— 排程

可选设置 **Frozen until(冻结至)**:在该时间点之前不允许任何工序开始,适合锁定
近期已定的工作。点 **Run scheduler**。结果会显示 **makespan** 与是否
**feasible(可行)**。

若不可行,说明启用资源无法满足某项能力、技能、资质或可用性 —— 调整资源或工作流后
重跑。

## 第 4 步 —— 审核、发布、执行

- **Review & publish** 将版本从 Reviewed 推进到 Published,并让其分配变为可执行
  (Ready)。
- **Timeline(时间线)** 以按资源分组的条形图展示分配。
- 在 **Assignments(分配)** 表中逐行驱动执行:**Start** → **Complete**(或
  **Fail** / **Cancel**,需填写原因)。

---

# 7. 计划版本生命周期

版本单向推进:

```text
Working(工作中) → Scheduled(已排程) → Reviewed(已审核) → Published(已发布) → Archived(已归档)
```

- **Working** —— 刚创建,尚无被接受的排程。
- **Scheduled** —— 产出了可行排程。
- **Reviewed** —— 计划员已接受。
- **Published** —— 正式排程,不可修改,其分配变为可执行。每个计划只能有一个已发布
  版本。
- **Archived** —— 历史记录。

非法操作(如未审核就发布、或对已发布版本重排)会被拒绝并给出明确提示。

---

# 8. 执行分配

版本发布后,每个分配从 **Ready** 开始并推进:

```text
Ready → Running → Completed
             ↘ Failed(需填原因)
Ready  ↘ Cancelled(需填原因)
```

每次状态转换都会记录到审计历史(可按分配查询)。Completed / Failed / Cancelled
为终态。

# 9. 报表

- **Dashboard** —— 计划、版本、已发布版本、设备、人员与工作流的总数。
- **KPI**(通过报表 API)—— 分配状态分布,以及各设备的工作量(分配数量与总占用
  时长)。

# 10. 角色

- **Production LM(生产负责人)** —— 日常计划:计划、版本、排程、执行。
- **Administrator(管理员)** —— 以上全部,外加创建用户 / 签发令牌。

权限由 API 强制;若你没有某操作所需角色,会收到 "禁止(forbidden)" 提示。

# 11. 常见问题排查

| 现象                              | 可能原因 / 处理                                                                      |
| --------------------------------- | ------------------------------------------------------------------------------------ |
| 提示需要有效认证令牌              | 未登录或令牌失效 —— 重新登录。                                                       |
| 排程结果 **不可行(infeasible)**   | 没有启用资源满足某能力 / 技能 / 资质,或可用性 / 冻结窗留不出空间。检查资源与工作流。 |
| 某工序因资质无法分配              | 该人员资质已过期 —— 在 Staff 页更新到期日。                                          |
| 已停用的设备/人员仍出现在旧排程里 | 快照按设计不可变;停用只影响后续排程。                                                |
| 点 "Run scheduler" 无反应         | 请先生成实例(第 1 步),空版本无法排程。                                               |

# 12. 关于时间与单位

本版本在计划周期内以抽象的整数时间单位排程(非真实日历日期)。时长、可用时间窗和
冻结边界都使用同一单位。真实日历日期与班次为后续计划的增强项。
